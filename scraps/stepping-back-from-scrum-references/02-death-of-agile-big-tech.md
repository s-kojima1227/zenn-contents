## 書き起こし

それはタイマーから始まった。
10人が輪になって立ち、「昨日・今日・ブロッカー」を順番に言わされていた。誰も顔を上げなかった。スクラムマスターのストップウォッチだけが、誰の声よりも大きく刻んでいた。
毎回の更新は前日の繰り返し——何も動かない、何も変わらない、誰も気にしない。
その瞬間、私は悟った。これはチームワークではない。これは演劇だ。そして静かに、優秀なエンジニアたちがすでにスクラムを廃止した会社へと抜けていった。

### アジャイルがアジャイルだった頃（そしてなぜ機能したのか）

はっきり言っておこう。アジャイルが登場したとき、それは革命だった。
- 重厚なウォーターフォールはもうない
- 現実に触れた瞬間に崩れ去る、何ヶ月もかけた事前計画もない
- リアルなコラボレーション。リアルなフィードバック

あの初期の頃、私たちは速くものを作り、価値を届け、走りながら学んでいた。
しかし「アジャイル」がブランドになったとき、何かが変わった。ムーブメントとして始まったものが、**カーゴカルト（形だけの模倣）**に成り下がった。

### スプリントセレモニーの隠れたコスト

自分のチームをよく見てほしい。週に何時間をここに費やしているか？

| セレモニー | 平均時間 | 8人チームでの合計 |
|-----------|---------|----------------|
| スタンドアップ | 15分 | 週1時間 |
| プランニング | 1時間 | 週1時間 |
| グルーミング | 1時間 | 週1時間 |
| レトロ | 1時間 | 週1時間 |
| デモ | 1時間 | 週1時間 |
| **合計** | | **週5時間** |

8人チームなら週40時間——毎週エンジニア1人分が会議に消えている。ボードの準備、チケット更新、ステータス同期に費やす時間はこれにすら含まれていない。ビッグテックが「これで何を得ているのか？」と問い始めたのは当然だ。

### 本当の殺し屋：プロセス負債

問題は会議だけではない。プロセス負債だ。
チケットを動かし、「完成の定義」チェックリストを更新し、見積もりを正当化することが本来の仕事になったとき、職人としての技術は死ぬ。エンジニアはオーナーシップを失う。仕事は成果ではなく、見た目のために行われるようになる。

「アジャイル」ワークフロー：
```
[チケット] ←ループ
     ↓
[スプリント計画]
     ↓
[デイリースタンドアップ]
     ↓
[進行中/ブロック]
     ↓
[コードレビュー]
     ↓
[完了/QA]
     ↓
[レトロ]
```

各ステップが、実際のプロダクト価値からあなたを引き離す。

### コードが真実を語る

プロセス負債がアジリティを殺す様子を、実例で見せよう。

問題：シンプルな機能が何日もブロックされる。「パスワードリセット」エンドポイントを追加する必要があった。コードはこれだけ：

```java
@PostMapping("/reset-password")
public ResponseEntity<Void> resetPassword(@RequestBody ResetPasswordRequest req) {
    userService.resetPassword(req.getEmail());
    return ResponseEntity.ok().build();
}
```

しかしスクラムの世界ではこうなった：
- チケット作成（「ユーザーストーリーと受け入れ条件を書け」）
- グルーミングに追加。来週のセッションまで待つ
- ストーリーポイントの議論：1か2か3か？（チームの半分が聞いていない）
- プランニング。スプリント開始まで待つ
- ブロック：APIスペックのレビューが必要。もう一週間待つ
- ようやく実装——15分のコード作業

アイデアからデプロイまで：3週間。

変化：チケットを殺し、チームを信頼する。ある金曜日、実験をした。小さな変更にチケットなし。エンジニアがプロダクトから直接リクエストを受け取り、ピアレビューの後にそのままtrunkへship。

結果：
- 本番環境へのリードタイムが21日から1日に短縮
- バグ報告ゼロ
- チームのモラルが急上昇

### ビッグテックの静かなピボット：スクラムを超えて

Amazon・Netflix・Meta・Googleに聞いてみるといい。廊下でこんな声が聞こえてくる：
- 「アジャイルはまだやってるけど、教科書通りじゃない」
- 「カンバンか、フレームワークなしか」
- 「スプリントが終わるときではなく、準備ができたときにshipする」

最高のチームは静かに進化した：
- セレモニーを減らし、自律性を高める
- エンジニアとステークホルダーの直接アクセス
- 自動デプロイ、即時フィードバック
- プロセスではなく、オーナーシップを中心に置く

### アジャイルが殺したもの——そして復活させるべきもの

大組織における「アジャイル」の悲劇は、失敗したことではなく、演劇になったことだ。
すべてのスタンドアップ、すべてのレトロ、すべてのJIRAの儀式を守り続けているなら、問いかけてほしい——チームは速くなっているか？ 幸せか？ より頻繁にshipしているか？

私が見てきた最高のチームのアーキテクチャはこれだ：

```
プロダクトステークホルダー
        ↓
    エンジニア
        ↓
   カスタマー
```

中間にプロセスの壁はない。アイデアからコードからユーザーへの直接ライン。

### 本当のアドバイス：プロセスを見えなくせよ

私が今すべてのチームに伝えていること：
- プロセスを最小化する。できることは自動化する
- エンジニアが自分の仕事をエンドツーエンドで所有する
- 直接コミュニケーション——エンジニアからPM、PMからユーザーへ
- 完了したストーリーポイントではなく、デリバリーした価値を計測する

儀式が助けになっていないなら、殺せ。アジャイルは死んでいない。カーゴカルトが死んだのだ。最高のチームはその違いを知っている。

### 最後に：スクラムを悼むな——もっと良いものを作れ

スクラムに葬式は要らない。必要なのはエンジニアリングへの信頼の再生だ。
ビッグテックは壁の文字を読んだ。最も賢いチームはアジリティを捨てなかった——プロセスの鎖を断ち切ることで、アジリティを再発見した。
同じことをせよ。あなたの最高のコードは、その向こう側で待っている。
