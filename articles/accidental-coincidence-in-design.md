## 🤔 そのフィールド、何役兼ねていますか？

`email` フィールドを、ログインの認証識別子としても、画面の表示名としても、通知の送信先としても使っている。よくある構成です。

でも「表示名をニックネームに変えたい」と言われた瞬間、困ります。3つの異なる役割が1つのフィールドに同居していて、**たまたま同じ値だったから問題にならなかっただけ**だったと気づきます。

多くのコードベースで、**値が同じであること**と**意味が同じであること**が静かに混同されています。そしてその混同は、要件が変わるまで気づきにくいです。

## 🔍 偶然の一致と本質的な一致

2つの用語を整理します。

- **本質的な一致**: 2つの値が同じなのは、ドメイン上「同じであるべき」だから。一方が変われば他方も変わるのが自然
- **偶然の一致**: 2つの値がたまたま今は同じだが、ドメイン上は独立した概念。一方だけが変わっても何らおかしくない

厄介なのは、コード上でこの2つが同じに見えることです。同じフィールドを参照し、同じ変数を使い、同じ関数を通る。値が一致している限り、区別する動機が生まれにくいです。

## 📦 よくある例

メールアドレス以外にも、実務で頻出するパターンがあります。

### 請求先住所と配送先住所

ECサイトで `address` フィールドが1つだけある。最初のユーザーは大抵自宅に届けて自宅で払うので問題ありません。「会社に届けてほしい」「請求書は本社宛で」と言われると対応が難しくなります。

### 「表示用の値」と「計算用の値」

商品の税込価格を画面に表示し、同じ値で集計処理もしている。ある日「画面にはキャンペーン価格を出したいが、集計は通常価格で行いたい」という要件が来ます。1つの `price` フィールドが「表示価格」と「計算基準価格」を兼ねていたことに、ここで初めて表面化します。

### 処理の偶然の一致

値だけでなく、**処理**にも同じ罠があります。「ユーザー登録時のバリデーション」と「プロフィール更新時のバリデーション」が今は同じロジックだからといって、共通関数 `validateUser()` にまとめる。後から「登録時は招待コード必須だが更新時は不要」となったとき、共通化したはずの関数に条件分岐が増殖していきます。

## 🙈 なぜ気づけないのか

この問題がやっかいなのは、**あらゆるフィードバックが「問題なし」と言ってくる**ことです。

### テストが通る

値が同じである限り、概念を分けていてもいなくても動作は変わりません。テストは「今の仕様に対する正しさ」を検証するものなので、この混同にはテストからは気づきにくいです。

### コードレビューで褒められる

フィールドが少ない。関数が共通化されている。DRYに見える。レビュアーの目には「シンプルで良い設計」と映ります。実際、今この瞬間の仕様に対してはシンプルで正しいです。問題は、そのシンプルさの一部が設計判断ではなく偶然の産物であることが、外から見えない点です。

### DRY原則が誤った確信を与える

「重複を排除せよ」は広く知られた原則です。しかし DRY が排除すべきは**知識（knowledge）の重複**であって、**値やコードの文字面の重複**ではありません。別々の概念がたまたま同じ値を持っているとき、それを1つにまとめるのは DRY ではなく、**異なる知識の結合**になってしまう可能性があります。

### 命名が解像度を下げる

`email` という命名は間違っていません。しかし正確でもありません。「認証識別子としてのメールアドレス」「表示名としてのメールアドレス」「通知先としてのメールアドレス」のどれなのかは、`email` という名前からは読み取れません。汎用的な命名は、概念の多義性を隠してしまいます。

## 💡 どう気づくか

万能な手法はありませんが、いくつかの問いかけが助けになります。

### 「この値や処理が別々に変わることはあるか？」

最も直接的な問いかけです。1つのフィールドや関数を見て「これが参照される場所ごとに、独立して変わる可能性はないか？」と考えます。変わりうるなら、今は同じでもそれは偶然です。

これはYAGNI（必要になるまで作るな）と矛盾しません。YAGNIは「機能を先回りして作るな」という原則であって、「概念の区別をサボれ」という原則ではありません。

### ドメインエキスパートの言葉を確認する

ドメインエキスパートと会話するときに「メールアドレス」ではなく「ログイン用のメールアドレス」「通知先」「表示名」と言い分けてみます。相手が「ああ、それは別の話だね」と反応したら、概念として別物であるサインかもしれません。

### 参照箇所を列挙する

あるフィールドが参照されている箇所を全部洗い出し、参照元ごとに「なぜこの値を使っているのか」を言語化してみます。理由が複数出てきたら、複数の役割が1つのフィールドに同居しているサインです。

## 🛠️ 気づいたあとの実装方針

偶然の一致を発見したとして、どう対処するか。段階的なアプローチを示します。

### まず概念として分離し、値は共有する

最も穏当な第一歩は、**データは1つのまま、振る舞い（メソッド）で役割の違いを表現する**ことです。

```go
type User struct {
    Email Email
}

// 認証識別子としてのメールアドレス
func (u *User) LoginIdentifier() Email {
    return u.Email
}

// 表示名
func (u *User) DisplayName() string {
    return string(u.Email)
}

// 通知先
func (u *User) NotificationDestination() Email {
    return u.Email
}
```

この時点ではデータ構造は何も変わりません。しかし「このメールアドレスは3つの異なる役割で参照されている」ことがコード上で表現されています。参照する側は `user.Email` ではなく `user.DisplayName()` や `user.NotificationDestination()` を使うことで、意図が明確になります。

データが1つのままなので、整合性の問題も起きません。

### 実際に分離が必要になったら、フィールド・カラムを分ける

「表示名をニックネームにしたい」という要件が実際に来たとき、初めてフィールドとデータベースカラムを分離します。

```go
type User struct {
    Email       Email
    DisplayName string // 新しく追加
}

func (u *User) LoginIdentifier() Email {
    return u.Email
}

func (u *User) DisplayName() string {
    return u.DisplayName // 独立したフィールドを返すように変更
}

func (u *User) NotificationDestination() Email {
    return u.Email
}
```

データベースにも `display_name` カラムを追加し、リポジトリ層でマッピングします。

```go
func (r *UserRepository) toEntity(row *UserRow) *User {
    return &User{
        Email:       Email(row.Email),
        DisplayName: row.DisplayName,
    }
}
```

振る舞いで参照していたおかげで、変更の影響は `DisplayName()` メソッドの内部とリポジトリ層に閉じます。呼び出し側のコードは変わりません。

### 処理の場合も考え方は同じ

「よくある例」で挙げたバリデーションのケースも同じアプローチが使えます。まず別の関数として定義し、中身は同じ処理を呼びます。

```go
func validateForRegistration(input RegistrationInput) error {
    return validateCommon(input.Email, input.Name)
}

func validateForUpdate(input UpdateInput) error {
    return validateCommon(input.Email, input.Name)
}

func validateCommon(email Email, name string) error {
    // 共通のバリデーション
}
```

今は `validateForRegistration` と `validateForUpdate` の中身が同じでも、「登録時のバリデーション」と「更新時のバリデーション」は概念として別であることが表現されています。

要件が分かれたら、固有のロジックを追加するだけです。

```go
func validateForRegistration(input RegistrationInput) error {
    if err := validateCommon(input.Email, input.Name); err != nil {
        return err
    }
    // 登録固有：招待コード必須
    if input.InviteCode == "" {
        return errors.New("invite code is required")
    }
    return nil
}
```

値の場合と同様に、概念を分けておいたおかげで、変更の影響が `validateForRegistration` に閉じます。

## 📝 まとめ

- 値や処理が同じでも、概念として同じとは限らない
- テストもレビューもこの混同を指摘してくれない。「この値や処理が別々に変わることはあるか？」と自分で問いかける
- 気づいたら、まず振る舞いで役割の違いを表現する。データやカラムの分離は実際に必要になってから

すべてを先回りして分離する必要はありません。ただ「この一致は偶然かもしれない」と意識しているだけで、変更が来たときの初動が変わると思います。
