## 🤔 要件の確定を待っていたら、間に合わない

私たちのチームでは、タイトなスケジュールで機能開発をする場面がちょくちょくあります。先日もまさにそういう場面がありました。要件の整理やデザインの完成を待ってから開発に着手していては、どう考えてもスケジュールに間に合わない。かといって、何も決まっていない状態で闇雲にコードを書けば、手戻りの嵐になります。

しかもリソースの都合で、この機能は自分ひとりで進める必要がありました。

そこで、**確定していることと未確定なことを見極めて、確定部分から段階的に肉付けしていく**という進め方をしました。自分ひとりで進めるので進め方は自由に決められる一方、レビューは他のメンバーに見てもらうので、メンバーにも要件や設計を理解してもらいつつ、レビュー負荷を高めないことも意識していました。

特別なことをしているわけではないですが、今回なんとかやりきれたので、具体的にどう進めたかを改めて言葉にしてみます。

## 🔍 確定していることと、未確定なこと

要件が固まっていない段階でも、すでに確定していることは意外とありました。

ここからは「商品にレビュー機能をつける」という架空の機能を例に説明します。要件の詳細は未定でも、以下のようなことはほぼ確定していました。

**確定していること（構造）**
- 「レビュー」というエンティティが存在すること
- レビューは「ユーザー」と「商品」に紐づくこと
- レビューの一覧を取得するAPIが必要なこと
- レビューを投稿するAPIが必要なこと

**未確定なこと（詳細）**
- レビューにどんな項目があるか（星の数？テキスト？画像？）
- 一覧のソート順やフィルタ条件
- 投稿時のバリデーションルール

エンティティが何であるか、エンティティ同士の関係性はどうか、といった**構造は確定的**です。一方、エンティティの詳細なフィールドや振る舞いは**要件次第**で変わります。

この境界を意識したことで、要件が固まる前から着手できる範囲が見えてきました。

## 💡 段階的に肉付けしていく

この見極めをもとに、以下のように段階的に開発を進めました。全体の流れを先に示します。

```
Phase 1：最低限必要なAPIを定義する（スキーマ＝設計）
Phase 2：バックエンドに骨組みを作る（エンティティ・ハンドラ・疑似コード）
Phase 3：ウォーキングスケルトンで縦を通す（フロント〜バックエンド一気通貫）
Phase 4：受け入れ条件をチームでレビューする（仕様の抜け漏れを潰す）
Phase 5：本実装する（デザイン適用・AIコーディング）
```

ここからは各フェーズの詳細です。

### Phase 1：最低限必要なAPIを定義する

まず、要件を満たすために**最低限これは必要だろうと思えるAPI**を洗い出して定義しました。この時点では詳細な項目はまだ埋めず、リクエスト・レスポンスにどのような情報が含まれそうかを定性的なコメントで残しておきました。

私たちのチームではgRPCを使っているので、以下のようなスキーマになります。GraphQLやOpenAPIなど、スキーマ駆動の技術スタックであれば同じアプローチが取れると思います。

```protobuf
service ReviewService {
  // 商品に対するレビューを投稿する
  rpc PostReview(PostReviewRequest) returns (PostReviewResponse);

  // 商品に紐づくレビュー一覧を取得する
  rpc ListReviews(ListReviewsRequest) returns (ListReviewsResponse);
}

message PostReviewRequest {
  string product_id = 1;
  // 評価（段階評価のようなもの）
  // テキストによるレビュー本文
  // 画像の添付があるかもしれない
}

message PostReviewResponse {
  string review_id = 1;
}

message ListReviewsRequest {
  string product_id = 1;
  // ページネーション
  // ソートやフィルタの条件が入るかもしれない
}

message ListReviewsResponse {
  // レビューのリスト
  // レビューには投稿者情報、評価、本文などが含まれる
}
```

このスキーマが**設計の代わり**になりました。APIの一覧とコメントを見れば、この機能が何をするのか、どんなデータがやり取りされるのかの大枠が伝わります。PRを通じて開発チーム全体にその機能の大まかな理解が共有されました。

### Phase 2：バックエンドに骨組みを作る

Phase 1で定義したスキーマをバックエンドに取り込み、最低限のエンティティの骨組みと関係性を実装しました。DBのテーブルやマイグレーションもこの段階で用意しました。

```go
// entity.go
type Review struct {
    ID        string
    UserID    string
    ProductID string
    CreatedAt time.Time
}
```

```go
// repository.go
type ReviewRepository interface {
    Save(ctx context.Context, review *Review) error
    FindByProductID(ctx context.Context, productID string) ([]*Review, error)
}
```

そしてRPCのハンドラの骨組みを作りました。この段階でも、1つ1つのRPCに対して**疑似コードで処理の流れ**を書いていきました。

```go
func (h *ReviewHandler) PostReview(ctx context.Context, req *pb.PostReviewRequest) (*pb.PostReviewResponse, error) {
    // 1. リクエストのバリデーション

    // 2. 対象の商品が存在するか確認する

    // 3. レビューエンティティを生成する

    // 4. レビューを保存する

    // 5. 商品の出品者にレビュー投稿を通知する

    // 6. レスポンスを返す

    return &pb.PostReviewResponse{}, nil
}
```

この疑似コード付きの空実装をPRに出すことで、各RPCのざっくりした設計がチームに共有されました。PRを細かく切りながら、フィードバックを早く得る方法については、以下の記事で詳しく書いています。

https://zenn.dev/soycha/articles/pseudocode-first-development

### Phase 3：ウォーキングスケルトンで縦を通す

Phase 2で骨組みができたら、次はAPIの縦を通しました。エンティティはほとんどIDだけの状態のまま、フロントエンドからバックエンドまで一気通貫で操作できる状態を作りました。

フロントエンドはこの時点では疎通確認のための簡素な画面でした。例えばこんな感じです。

*商品一覧：商品IDが並んでいるだけ*

![レビュー一覧](/images/incremental-development-under-uncertainty/walking-skeleton-review-list.jpg)
*レビュー一覧：ID・ユーザーID・コメントが表示されるだけ*

![レビュー投稿](/images/incremental-development-under-uncertainty/walking-skeleton-review.jpg)
*レビュー投稿：フォーム1つとボタンだけ*

見た目は何もありません。でも、フロントからAPIを叩いてデータが保存され、一覧に反映されるという**一連の流れが動く**ことが重要です。

画面はこの程度ですが、API呼び出しのフックやページの骨組みはデザインに依存しにくいので、この段階でちゃんと作っておきました。これが後の本実装で効いてきます。

縦を通す過程で、連携先がまだ準備できていないケースも出てきました。疑似コードのStep 5「通知する」で呼び出す通知サービスが、別チームでまだ開発中だったのです。この場合は、インターフェースだけ定義してモック実装で通しました。

```go
type ReviewNotifier interface {
    NotifyNewReview(ctx context.Context, review *Review) error
}

// 通知サービスが開発中のため、モックで縦を通す
type mockReviewNotifier struct{}

func (m *mockReviewNotifier) NotifyNewReview(ctx context.Context, review *Review) error {
    log.Printf("mock: notify new review %s", review.ID)
    return nil
}
```

通知サービスが完成したら、モックを本実装に差し替えるだけです。連携先の完成を待たずに、自チームの一連の流れを先に動かせました。

この段階で縦を通しておいたことで、後のフェーズで詳細を肉付けしていくときに、動作確認のベースがすでにあるので開発効率が上がりました。

### Phase 4：受け入れ条件をチームでレビューする

基本的には、ここまで進めている間に要件が固まり、デザインが上がってきました。

要件や基本的な仕様はPdMがしっかり決めてくれていました。ここからはそれをユースケース単位でチケットに細かく切り、それぞれに受け入れ条件を書いていきました。PdMが決めた仕様を、よりミクロな視点で「このユースケースで考慮漏れはないか？」と詰めていく作業です。

例えば「レビューを投稿する」のチケットなら、こんな受け入れ条件になります。

> **レビューを投稿する**
>
> - 商品に対して、評価（1〜5）・タイトル・コメントを入力してレビューを投稿できる
> - 画像を最大3枚まで添付できる
> - タイトルは必須、コメントは1000文字以内
> - 投稿後、レビュー一覧に即時反映される
> - 投稿後、商品の出品者に通知が送られる

実装に入る前に、この受け入れ条件をチームでレビューしました。ここで指摘が入ります。

- 「同じユーザーが同じ商品に複数回レビューできる？1回だけ？」
- 「画像のファイル形式やサイズの制限は？」
- 「自分が出品した商品にレビューできちゃわない？」

こうしたフィードバックが**実装前に**入ります。これを受けて、受け入れ条件はこう更新されました。

> **レビューを投稿する（レビュー後）**
>
> - 商品に対して、評価（1〜5）・タイトル・コメントを入力してレビューを投稿できる
> - 画像を最大3枚まで添付できる（JPEG/PNG、1枚5MB以内）
> - タイトルは必須、コメントは1000文字以内
> - **同じ商品へのレビューは1ユーザー1回まで**
> - **自分が出品した商品にはレビューできない**
> - 投稿後、レビュー一覧に即時反映される
> - 投稿後、商品の出品者に通知が送られる

受け入れ条件を直すだけなので、コストはほぼゼロです。仕様の抜け漏れが潰され、手戻りを減らせるだけでなく、サービスの品質も上がりました。

### Phase 5：本実装する

Phase 4のレビューを経たら、確定した要件に合わせてスキーマにフィールドを追加し、エンティティやバリデーションを本実装していきました。

本実装でもPRはなるべく小さくしたかったので、ユースケースをさらにステップに分割してサブタスク化しました。例えば「レビューを投稿する」なら、こんな感じです。

```
Step 1: 投稿ボタン + 空のモーダル表示
Step 2: 評価・タイトル・コメントのフォーム + 画像選択UI
Step 3: バリデーション（入力チェック・ファイルサイズ制限）
Step 4: 投稿処理（API呼び出し + 一覧への反映）
```

1ステップずつPRを出していくので、レビューも軽く、マージも速かったです。

最終的に、スキーマとエンティティはこうなりました。Phase 1の時点ではコメントだけだったフィールドが、すべて埋まっています。

```protobuf
message PostReviewRequest {
  string product_id = 1;
  int32 rating = 2;        // 5段階評価
  string title = 3;        // タイトル
  string comment = 4;
  repeated string image_urls = 5; // 画像
}
```

```go
type Review struct {
    ID        string
    UserID    string
    ProductID string
    Rating    int
    Title     string
    Comment   string
    ImageURLs []string
    CreatedAt time.Time
}

func (r *Review) Validate() error {
    if r.Rating < 1 || r.Rating > 5 {
        return errors.New("rating must be between 1 and 5")
    }
    if r.Title == "" {
        return errors.New("title is required")
    }
    if len(r.Comment) > 1000 {
        return errors.New("comment too long")
    }
    return nil
}
```

Phase 1〜3で構造・骨組み・縦の流れがすでに動いているので、この段階での作業はフィールドの追加やバリデーション、フロントエンドのデザイン適用が中心でした。アーキテクチャの変更にはなりませんでした。

私たちのチームではデザインをFigmaで管理しているので、フロントエンドの本実装はFigma MCPを使ったAIコーディングで進めました。ウォーキングスケルトンの段階でAPI呼び出しのフックやページの骨組みがすでにあるので、AIはその既存コードとFigmaのデザインを比較しながらデザインを適用していくだけです。さらに受け入れ条件も詳細に書いてあるので、AIの実装精度がかなり高かったです。

テストもPlaywright MCPを使って、受け入れ条件をもとにAIに書かせました。受け入れ条件が具体的に書いてあるので、そのままテストケースに落とし込んでくれました。これもかなり快適でした。

ここまでの段階的な積み上げがあったおかげで、本実装のフェーズは想像以上に楽でした。

## ✅ この進め方でよかったこと

### 小さくはじめて、不確実性を段階的に減らせた

確定していることと未確定なことを見極めて、確定部分から小さく進めたことがすべての土台でした。スキーマだけ、骨組みだけ、縦を通すだけ——各フェーズでやることを絞り、1つずつPRにして進めました。

不確実性がフェーズごとに少しずつ減っていくので、「あとは要件が決まればフィールド追加とバリデーション調整だけ」と早い段階で言えるようになりました。PdMが要件を詰めている間にスキーマ定義と骨組みを作り、デザイナーがデザインを仕上げている間にウォーキングスケルトンで縦を通す。手が止まる時間がほとんどなく、チーム全体のスループットが上がりました。

### どのフェーズでもフィードバックが入り、手戻りが減った

スキーマ定義、疑似コード、ウォーキングスケルトン、受け入れ条件——どのフェーズにも、チームからフィードバックを得るポイントがありました。すべてPRやチケットを通じてレビューされるので、問題が実装前に見つかります。受け入れ条件を直すだけならコストはほぼゼロですが、実装後に仕様の抜け漏れが見つかれば手戻りは大きくなります。フィードバックループが前に来る仕組みを意識的に作ったことが、手戻りの少なさにつながりました。

また、フィードバックの機会が何度もあるということは、メンバーが要件について考える機会もそれだけ多いということです。フェーズが進むにつれて、チーム全体の要件理解が自然と深まっていきました。

### 段階的な積み上げが、AIコーディングの精度を引き出した

基本的にどのフェーズでもAIコーディングを活用していましたが、前のフェーズの成果物があるかないかで精度がかなり変わりました。疑似コードがあればAIはウォーキングスケルトンを書きやすいし、骨組みと受け入れ条件が揃っていればデザイン適用やテスト生成の精度も上がります。

段階的に積み上げていく過程で、疑似コードや受け入れ条件といったAIへの指示が自然と準備されている状態になっていました。だから指示もしやすいし、意図通りのものを出してくれる。設計書をまとめて渡していきなり作らせるよりも、AIへのフィードバックが少なく済んだ印象があります。

## 📝 おわりに

今回書いた進め方は、自分がなんとなくやっていたことを改めて言語化したものです。チームで「こうやろう」と決めてやっているわけではないし、まだまだ洗練されていない部分も多いと思います。でも、こうやって言葉に起こして共有することで、チームの進め方として磨いていけたらいいなと思っています。
